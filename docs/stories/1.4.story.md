# Story 1.4: Journal d'audit — écriture transactionnelle et immuabilité

## Status

Ready for Review

## Story

**En tant que** direction,
**je veux** que toute action sensible soit enregistrée de façon inaltérable,
**afin de** disposer d'une trace opposable, y compris en cas de litige entre associés.

## Acceptance Criteria

1. Une entrée porte : auteur (nullable pour l'amorçage), libellé d'auteur, horodatage, type d'objet, identifiant d'objet, action, ancienne valeur, nouvelle valeur, identifiant de requête.
2. `Support\Auditing\AuditLogger` et le trait `Auditable` écrivent **dans la transaction de l'opération métier**. ⛔ Un test provoque l'échec de l'écriture d'audit et vérifie que **l'opération métier est annulée** (NFR21).
3. **Triple barrière d'immuabilité** (A-05), les trois posées **par migration** : privilèges SQL (`INSERT` seul sur `audit_logs` pour l'utilisateur applicatif), déclencheurs base refusant `UPDATE` et `DELETE`, garde applicative dans le modèle.
4. ⛔ Aucune route, aucun formulaire, aucune commande ne modifie ni ne supprime une entrée ; testé pour `PUT`, `PATCH` et `DELETE`.
5. Une tentative d'`UPDATE` ou de `DELETE` directe en SQL sous l'utilisateur applicatif échoue ; testé sur MySQL.
6. Un observateur de filet de sécurité journalise toute écriture sur un modèle audité qui aurait contourné le service.
7. La table est indexée sur (type d'objet, identifiant d'objet), sur l'auteur et sur l'horodatage — la consultation filtrée de 2.10 en dépend.

**Migrations :** `audit_logs` **en premier**, avec ses déclencheurs et ses privilèges.

### Triple barrière d'immuabilité

- Barrière SQL de privilèges : l'utilisateur applicatif possède `SELECT` et `INSERT` sur `audit_logs`, jamais `UPDATE` ni `DELETE`. Le terme « INSERT seul » de l'AC signifie seul droit de mutation ; `SELECT` reste nécessaire à la consultation future de `direction`. [Source: architecture/14-journal-daudit-non-modifiable.md#141-trois-barrières--a-05]
- Barrière SQL de déclencheurs : `BEFORE UPDATE` et `BEFORE DELETE` exécutent `SIGNAL SQLSTATE '45000'`. Ils bloquent même un correctif manuel lancé avec un compte plus privilégié. [Source: architecture/14-journal-daudit-non-modifiable.md#141-trois-barrières--a-05]
- Barrière applicative : le trait `Immutable` du modèle refuse mise à jour et suppression. Cette troisième barrière ne peut matériellement pas être « posée par migration »; conformément à l'architecture, les privilèges et déclencheurs sont créés par migration, tandis que la garde est livrée en PHP sur le modèle. [Source: architecture/14-journal-daudit-non-modifiable.md#141-trois-barrières--a-05] [Source: architecture/15-immuabilit-historiques-et-annulations.md#153-application]
- La migration utilise `DB::unprepared()` pour les instructions MySQL spécifiques. C'est une exception explicitement prescrite à la préférence Eloquent du standard de code. [Source: architecture/database-schema.md#règles-non-négociables]

### Écriture transactionnelle

- `App\Support\Auditing\AuditLogger` est appelé explicitement dans les services métier et constitue le mécanisme principal. L'observateur n'est qu'un filet de sécurité, car il voit le changement technique mais pas l'intention ni le motif. [Source: architecture/14-journal-daudit-non-modifiable.md#142-écriture]
- L'audit n'est jamais mis en file ou différé. Il s'exécute dans la transaction métier, et toute exception remonte pour annuler l'opération. [Source: architecture/14-journal-daudit-non-modifiable.md#142-écriture] [Source: architecture/12-transactions-financires-atomiques.md#122-ce-que-garantit-ce-motif]
- Aucun modèle métier soumis à FR21 n'existe encore. Le test de transaction et celui de l'observateur utilisent donc des fixtures isolées, sans introduire de table métier de production avant son epic.
- FR21 couvrira : opérations financières, objectifs validés et modifications, comptes et changements d'état, attributions de rôles/permissions, documents et versions, exports sensibles. [Source: prd.md#53-journal-daudit--étape-1]

### Trait `Auditable` et observateur

- Le trait `Auditable` sert à déclarer et raccorder les futurs modèles soumis à audit. L'observateur placé sous `app/Observers/` garantit qu'une écriture ayant contourné le service explicite laisse tout de même une trace. [Source: architecture/14-journal-daudit-non-modifiable.md#142-écriture] [Source: architecture/source-tree.md#arborescence]
- Le mécanisme doit distinguer une écriture explicitement auditée d'un contournement pour ne pas créer deux lignes pour la même action. L'architecture exige ce résultat mais ne prescrit pas l'API ou le mécanisme interne de déduplication.
- L'observateur ne doit jamais retarder l'écriture, l'envoyer en file ou avaler une erreur.

### API et surface HTTP

- Il n'y a pas d'API publique et aucune interface de mutation du journal. Aucun contrôleur, Form Request, page ou commande de modification/suppression n'est créé dans cette story. [Source: architecture/rest-api-spec.md#rest-api-spec] [Source: prd.md#53-journal-daudit--étape-1]
- La consultation et l'export réservés à `direction` arrivent en story 2.10. Aucun accès de lecture utilisateur n'est anticipé ici. [Source: architecture/14-journal-daudit-non-modifiable.md#143-lecture--fr23--d-04]
- Le test HTTP doit inspecter la table de routage pour les verbes `PUT`, `PATCH` et `DELETE`, sans figer prématurément l'URI de consultation future.

### Project Structure Notes

- Le dépôt ne contient encore aucun modèle métier, observateur ou dossier `app/Support/Auditing/`; ces emplacements sont explicitement prévus par l'architecture.
- La seule migration existante crée l'infrastructure des jobs. Elle précède nécessairement la nouvelle migration d'audit, mais ne viole pas la règle « audit avant toute table métier ».
- Le job CI MySQL utilise aujourd'hui un seul compte `staffptr_ci`. Il doit être étendu pour distinguer migration privilégiée et application restreinte sans affaiblir la protection de `main`.
- Écart AC/architecture : l'AC 3 dit que les trois barrières sont « posées par migration », alors qu'une garde modèle est du PHP. Résolution : la migration pose les deux barrières SQL et le modèle porte la troisième, conformément à l'architecture § 14.1.
- Écart de vocabulaire : « INSERT seul » désigne le seul droit d'écriture. L'architecture accorde aussi `SELECT`, indispensable à la consultation future, mais aucun `UPDATE` ni `DELETE`.
- Dépendance séquentielle : les vrais comptes MySQL séparés sont livrés en 1.5. La story 1.4 utilise des comptes éphémères équivalents en CI pour fournir immédiatement la preuve exigée.

### Debug Log References

- La première CI échouait au chargement de la migration car un `use RuntimeException` global devenait un avertissement bloquant en environnement de test ; la référence globale a été corrigée et formatée par Pint.
- MySQL 8 avec journal binaire refusait les déclencheurs au compte de migration sans privilège `SUPER` (erreur 1419) ; la CI active désormais `log_bin_trust_function_creators=1`, sans accorder de privilège global excessif au compte dédié.
- Deux assertions MySQL partageaient des traces immuables entre tests : la vérification de privilèges interroge maintenant les droits du compte applicatif lui-même et la déduplication compare le nombre d'audits automatiques avant/après l'opération explicite.

### Completion Notes List

- Migration `audit_logs` livrée avec toutes les colonnes brutes requises, précision UTC milliseconde et index de consultation ; aucune table `users` ni clé étrangère prématurée n'a été créée.
- Triple barrière validée : garde Eloquent `Immutable`, déclencheurs MySQL `BEFORE UPDATE/DELETE` et compte applicatif limité à `SELECT`/`INSERT` sur le journal.
- `AuditLogger::runExplicitly()` impose une transaction active, conserve le libellé d'auteur et l'identifiant de corrélation, et laisse remonter toute exception afin d'annuler l'opération métier.
- Le trait `Auditable` encapsule les écritures directes dans une transaction ; `AuditableObserver` journalise créations/modifications contournées et `AuditContext` évite le double audit explicite/automatique.
- Aucune route, commande, Form Request ou page de formulaire de mutation du journal n'est exposée ; les verbes `PUT`, `PATCH` et `DELETE` sont couverts séparément.
- CI séparée en comptes éphémères `staffptr_migrate_ci` et `staffptr_app_ci`. Exécution verte `29675237799` : 53 tests, 387 assertions sous MySQL 8, bundle 80,55 Ko et durée 79 s / 600 s.
- Validation locale : 54 tests, 47 réussis et 7 preuves MySQL ignorées hors CI, 268 assertions, migration SQLite fraîche, Pint, Larastan niveau 6, quatre tests Node et build Vite réussis.
- Checklist DoD exécutée : tous les éléments applicables sont satisfaits ; API publique, écran utilisateur, seeder de production et test E2E métier sont sans objet. Aucun reliquat technique connu dans le périmètre de la story.
