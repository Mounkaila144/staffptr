# Story 1.3: Chaîne d'intégration continue

## Status

Aprouved

## Story

**En tant qu'** équipe de développement,
**je veux** que la chaîne refuse automatiquement ce qui ne respecte pas les standards,
**afin que** la qualité ne dépende pas de la discipline d'un jour donné.

## Acceptance Criteria

1. Le workflow GitHub Actions s'exécute sur chaque pull request : `pint --test`, Larastan niveau 6, `phpunit`, `npm run build`.
2. **La suite tourne sur MySQL 8 en service Docker**, pas sur SQLite (DEC-02) : les déclencheurs d'immuabilité, les colonnes générées, les contraintes `CHECK` et `lockForUpdate()` n'existent pas ou diffèrent sous SQLite. Les tester sur un moteur qui ne les applique pas reviendrait à ne pas les tester.
3. Un **budget de poids** est vérifié : le bundle d'une page du parcours quotidien dépassant 300 Ko compressé fait échouer la chaîne (NFR2).
4. Une pull request dont un seul contrôle échoue ne peut pas être fusionnée ; la protection de branche est active sur `main`.
5. Playwright est installé et exécute au moins un parcours de démonstration ; la structure `tests/e2e/` existe.
6. La durée totale de la chaîne sur une pull request reste **sous 10 minutes**, mesurée et consignée.

## Dev Notes

### Previous Story Insights

- La story 1.2 est terminée avec 31 tests / 166 assertions, Pint propre, Larastan niveau 6 sans erreur et un build Vite réussi. La CI doit reprendre ces validations existantes sans réduire leur couverture. [Source: stories/1.2.story.md#Completion Notes List]
- La suite comprend maintenant des tests Unit purs, des tests Feature de contrats partagés et un garde-fou de migrations monétaires avec fixture invalide. Les tests de schéma doivent être exécutés sous MySQL dans la chaîne. [Source: stories/1.2.story.md#File List]
- `phpunit.xml` force actuellement SQLite en mémoire. La story 1.3 doit fournir une surcharge CI explicite vers MySQL et empêcher tout repli silencieux ; le confort SQLite local peut rester réservé aux tests rapides qui ne dépendent pas du moteur. [Source: stories/1.2.story.md#Debug Log References]

### Socle transverse applicable

- SOC-05 impose un test Feature par critère, MySQL en CI, Pint propre et Larastan niveau 6. SOC-09 impose un premier chargement inférieur ou égal à 300 Ko, les navigations suivantes sous 80 Ko et zéro ressource tierce à l'exécution. [Source: prd/socle-transverse.md]
- Les autres règles SOC restent opposables aux changements applicatifs, mais cette story porte principalement l'infrastructure de qualité et n'ajoute aucune route métier, écriture sensible ou écran métier. [Source: prd/socle-transverse.md]

### Contrat CI/CD

- Sur chaque pull request, la chaîne exécute `pint --test`, Larastan niveau 6, PHPUnit sur MySQL, `npm run build`, le budget de poids et Playwright. [Source: architecture/25-dploiement-https-secrets-et-cicd.md#256-cicd--github-actions]
- Une fusion dans `main` doit rester impossible dès qu'un seul contrôle obligatoire échoue. Les noms des contrôles doivent donc être stables et correspondre exactement aux règles de protection GitHub.
- Le déploiement en préproduction et la promotion manuelle en production sont décrits dans l'architecture, mais ne font pas partie des critères de cette story : la story 1.3 s'arrête à la validation des pull requests et à la protection de branche. [Source: architecture/25-dploiement-https-secrets-et-cicd.md#256-cicd--github-actions]
- La limite de dix minutes porte sur la durée totale observable de la chaîne d'une pull request, pas sur la somme arithmétique de jobs parallèles.

### MySQL 8 en CI

- MySQL 8 est obligatoire dans l'environnement de test CI. SQLite ne prouve pas les déclencheurs, colonnes générées, contraintes `CHECK` ni `lockForUpdate()`. [Source: architecture/tech-stack.md#avertissement--sqlite-ne-teste-pas-ce-qui-compte-dec-02] [Source: architecture/testing-strategy.md#base-de-test--mysql-pas-sqlite-dec-02]
- L'environnement CI utilise MySQL en service Docker, des files synchrones et des factories ; aucun jeu de données réel ni secret de production n'est requis. [Source: architecture/24-environnements.md#241-les-quatre-environnements]
- Les migrations doivent partir d'un schéma propre dans la CI. Une migration déployée n'est jamais modifiée pour accommoder le pipeline. [Source: architecture/database-schema.md#règles-non-négociables]
- Aucun modèle ou schéma métier supplémentaire n'est requis par cette story.

### Budget de poids

- Le premier rendu utile sous réseau 400 kbit/s / 400 ms vise moins de trois secondes ; le premier chargement est limité à 300 Ko transférés et une navigation suivante à 80 Ko. [Source: architecture/18-performance-sur-connexion-faible.md#181-budget-opposable]
- Le contrôle demandé par l'AC 3 porte sur le bundle compressé de la page du parcours quotidien. Comme ce parcours métier n'existe pas encore, la page `Platform/Demo` livrée en 1.1 sert de cible initiale et le mécanisme doit accepter les futures pages sans réécriture.
- L'architecture de production utilise Brotli avec repli gzip. La méthode de compression choisie par le contrôle doit être documentée et stable pour rendre les mesures comparables. [Source: architecture/18-performance-sur-connexion-faible.md#182-ce-que-larchitecture-apporte]
- Les polices utilisent la pile système et toutes les ressources sont servies par l'application ; aucune ressource tierce n'est autorisée. [Source: architecture/18-performance-sur-connexion-faible.md#182-ce-que-larchitecture-apporte] [Source: architecture/tech-stack.md#interdits]

### Playwright

- Playwright 1.x est l'outil E2E imposé. Les tests résident sous `tests/e2e/`. [Source: architecture/tech-stack.md#tech-stack] [Source: architecture/testing-strategy.md#pyramide]
- Les parcours critiques futurs sont le rapport quotidien bridé, l'approbation de dépense, la connexion avec changement imposé et l'encaissement vers les parts et la réserve. Ils ne doivent pas être simulés avant que leurs fonctionnalités existent. [Source: architecture/testing-strategy.md#e2e--playwright]
- Le premier test de cette story couvre uniquement la démonstration existante dans un navigateur réel. L'émulation 400 kbit/s / 400 ms doit être possible dans la structure, mais la recette sur téléphone réel reste distincte. [Source: architecture/testing-strategy.md#e2e--playwright]

### Protection de branche et dépendance externe

- La protection de `main` est un réglage du dépôt GitHub, pas un fichier local. Elle doit exiger une pull request et tous les contrôles de qualité de la story.
- Le dépôt local est désormais relié à `origin` sur `https://github.com/Mounkaila144/staffptr.git` et la branche active est `main`.
- La dépendance externe restante porte sur les droits et le réglage de protection GitHub. Elle ne bloque pas la création des fichiers CI, des scripts et des tests ; elle bloque en revanche le passage de la story à `Done` tant que la protection n'est pas activée et vérifiée.

### API, données et interface

- Aucune API publique, route applicative, contrôleur, modèle, migration métier ou intégration externe n'est nécessaire. [Source: architecture/rest-api-spec.md#rest-api-spec] [Source: architecture/external-apis.md#external-apis]
- La page Inertia de démonstration existante est seulement consommée par le test E2E et le contrôle de poids ; cette story ne crée pas un nouvel écran métier.
- Les secrets applicatifs et de déploiement ne doivent jamais être ajoutés au workflow. Les identifiants de la base CI sont éphémères ; les futurs secrets de déploiement resteront dans GitHub Actions Secrets. [Source: architecture/25-dploiement-https-secrets-et-cicd.md#255-secrets]

### File Locations

- `.github/workflows/` — workflow de pull request et contrôles obligatoires.
- `package.json` et `package-lock.json` — Playwright et scripts de build/E2E/budget nécessaires.
- `playwright.config.*` — configuration E2E du dépôt.
- `tests/e2e/` — parcours de démonstration et future structure des parcours critiques.
- `tests/Feature/` — tests des contrats CI et du budget applicables aux artefacts versionnés.
- `phpunit.xml` ou configuration d'environnement du workflow — surcharge MySQL explicite en CI, sans casser le profil local.
- Documentation du dépôt ou `docs/ops/` — preuve de protection, mesure de durée et méthode du budget. [Source: architecture/source-tree.md#arborescence] [Source: architecture/testing-strategy.md#pyramide]

### Technical Constraints

- PHP 8.3, Laravel 13, PHPUnit 12, Larastan 3 niveau 6, Pint 1, Node/Vite 8 et Playwright 1.x. [Source: architecture/tech-stack.md#tech-stack]
- Les dépendances sont installées depuis `composer.lock` et `package-lock.json`; les versions ne doivent pas dériver entre exécutions.
- Aucune ressource tierce n'est chargée par l'application à l'exécution. GitHub Actions, les téléchargements de dépendances et d'images de service appartiennent à l'environnement de construction, pas au runtime applicatif. [Source: architecture/tech-stack.md#interdits]
- Le workflow ne doit contenir aucun secret réel ni contourner les tests en fonction de la branche. [Source: architecture/25-dploiement-https-secrets-et-cicd.md#255-secrets]

### Testing

- PHPUnit uniquement pour les tests PHP ; tests Unit sous `tests/Unit/`, Feature sous `tests/Feature/`, HTTP sous `tests/Feature/Http/` et E2E sous `tests/e2e/`. [Source: architecture/testing-strategy.md#pyramide]
- Chaque AC doit être traçable vers une preuve automatisée ou, pour la protection GitHub et la durée réelle, vers une exécution externe consignée.
- Tester le workflow ou ses scripts sur les chemins d'échec : Pint en défaut, analyse statique en défaut, test PHP rouge, build en échec, mauvais moteur de base, bundle juste sous / égal / au-dessus du seuil et E2E en échec.
- Vérifier explicitement que MySQL 8 est le moteur de la suite d'intégration ; une chaîne verte sur SQLite ne satisfait pas l'AC 2.
- Le test Playwright ouvre un navigateur réel, attend le rendu de la démonstration et échoue sur erreur de page ou de console non attendue.
- La mesure de durée et la vérification de protection nécessitent une pull request GitHub réelle ; une simulation locale seule ne satisfait pas les AC 4 et 6.
- Avant soumission : `php artisan test`, `vendor/bin/pint --dirty`, Larastan niveau 6 et `npm run build`. Sur la pull request : `pint --test`, Larastan niveau 6, PHPUnit/MySQL, build, budget et Playwright. [Source: architecture/coding-standards.md#avant-de-finaliser] [Source: architecture/25-dploiement-https-secrets-et-cicd.md#256-cicd--github-actions]

### Project Structure Notes

- Le dépôt ne contient actuellement ni `.github/workflows/`, ni configuration Playwright, ni `tests/e2e/`.
- `package.json` expose seulement les scripts Vite et BMAD ; Playwright et le contrôle de budget ne sont pas installés.
- `phpunit.xml` utilise SQLite `:memory:`. La CI doit surcharger explicitement cette configuration pour la suite d'intégration MySQL.
- La branche locale active est `main` et le remote `origin` pointe vers `https://github.com/Mounkaila144/staffptr.git`. Il reste à configurer et prouver la protection de branche dans GitHub avant que la story puisse devenir `Done`.
- Aucun conflit d'arborescence : `.github/workflows/`, `tests/e2e/` et les fichiers de configuration à la racine sont conformes à la structure décrite. [Source: architecture/source-tree.md#arborescence]
